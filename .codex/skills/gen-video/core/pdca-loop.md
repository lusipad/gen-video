# PDCA Loop

`镜界.SKILL` 要成立，不能只有生成和采集，还必须有复核和回写。

这里定义的是这个技能最稳定的闭环规则：前台产出要走一轮 `PDCA`，后台学习也要走一轮 `PDCA`。

## 前台 PDCA

### Plan

在生成前先锁定：

- 用户目标
- 模型栈
- 执行平台
- mode
- 时长与交付物范围
- 真实性锚点与情绪目标

### Do

按当前任务生成：

- 剧本
- 分镜
- 素材资产表
- 图片提示词
- 视频提示词
- Flow 操作包或其他平台执行包

### Check

生成后不能直接结束，必须复核：

- 故事是否真的对准了用户目标
- 真实性锚点有没有漂移
- 连续性、节奏、高潮和收尾是否成立
- 提示词是否真的适配当前模型和平台
- 交付物是否足够让下一步执行，而不是只有一堆漂亮描述

这一层的最小产出应包括：

- `pass / fail`
- 关键风险点
- 证据或示例
- 必须立刻修的项
- 可以后续迭代的项

### Act

如果 `Check` 不通过，就继续修当前输出，不要把未过关的内容当完成。

如果 `Check` 通过，但暴露出系统性问题，则需要进一步动作：

- 补 benchmark
- 改 `profiles/`
- 改 `modes/`
- 改 `core/`
- 记入 `knowledge/log.md`

如果当前已经生成了成片，`Act` 不应只停在口头结论上。优先把审片 verdict 继续编译成 `benchmarks/video-review-actions.md/json`，再决定是局部重做镜头、补证据，还是写回长期知识。

## 后台 PDCA

### Plan

在学习前先定义：

- 当前要回答什么问题
- 这条新信息准备影响哪一层
- 它可能被写入 `wiki`、`profiles`、`core` 还是只留观察

### Do

后台脚本负责采集和整理信号，例如：

- nightly review
- issue inbox
- video learning digest
- query writeback
- tracked source refresh

### Check

学习知识也不能“抓到就算学会”，必须复核：

- 来源是否可信
- 信息是否真的新
- 是否和现有判断冲突
- 是否会改变执行
- 适合 `admit / defer / reject` 中哪一种

这一层的最小产出应包括：

- 审阅结论
- 证据
- 影响范围
- 推荐落点
- 下次复查时间或延后原因

### Act

根据 `Check` 的结果执行动作：

- `admit`
  更新 `wiki/`，必要时编译到 `profiles/`、`benchmarks/` 或 `core/`
- `defer`
  进入待观察队列，等更多证据
- `reject`
  记录原因，避免以后重复走同一条无效线索

所有动作都要有留痕，而不是只停在聊天或脑内判断里。

## 两条闭环的关系

- 前台 PDCA 负责保证“这次交付是否合格”
- 后台 PDCA 负责保证“下次判断会不会更准”

少了前台 `Check`，就会变成“生成完就算完成”。

少了后台 `Check`，就会变成“抓到新信息就当学会了”。

## 硬规则

- 没有 `Check`，不算闭环
- `Check` 不通过，不能直接结束
- `Act` 必须落到“修改输出”或“修改知识”之一
- benchmark 不只是评分面，也要承担 `Check -> Act` 的桥接作用
